# VibeVibe 严格交付流程（研发 → 上线）

> 依据：`C:\Users\70924\JohnsonWork\research_project\vibevibe\vibe-vibe\docs\Advanced` 全量文档
> 执行原则：每次迭代必须按此清单过一遍，缺一项要标记原因。

## 0) 立项与需求确认（先产品，后代码）
1. 先做想法验证（用户是谁、痛点是否刚需、是否愿意持续使用/付费）。
2. 与 AI 做需求澄清：让 AI 主动追问边界、异常、Out-of-Scope。
3. 产出 PRD（版本号、阶段、干系人、目标、范围、验收标准）。

## 1) 技术方案与项目初始化
1. 明确技术栈（前端/后端/数据库/第三方服务）并写入 README。
2. 建立项目规则（编码规范、目录、禁止项、依赖策略）。
3. 初始化仓库、分支策略、提交规范（避免直接在主干乱改）。

## 2) 开发阶段（Doc-driven + 小步快跑）
1. 按 PRD 拆任务（每个任务可独立验证）。
2. 后端优先定义数据模型/API 契约（含错误码、分页、鉴权）。
3. 前端按 API 契约联调；所有关键按钮加 loading/错误态/空态。
4. 安全基线：
   - 密钥只放 `.env` / 平台环境变量
   - 输入校验（后端兜底）
   - 权限检查（页面 + 接口双层）

## 3) 测试与质量门禁
1. 至少覆盖：
   - 健康检查
   - 认证流程
   - 核心业务流程
   - 参数校验/异常返回
2. 保证可复现构建：`build` 必过。
3. 回归检查：修 A 不能坏 B。

## 4) 文档与可交接性
1. README 必须可独立启动项目（新同学照着能跑起来）。
2. API 文档（OpenAPI）必须有中文 summary/description/examples。
3. 用户手册要“非技术可读”（流程化、图示化、低门槛文案）。

## 5) 发布上线（先稳，再快）
1. 发布前清单：环境变量、域名/证书、访问路径、日志可观测。
2. CI/CD：至少保证 Push/PR 自动构建与测试。
3. 上线策略：优先灰度/小流量验证，异常可快速回滚。

## 6) 上线后运营与迭代
1. 监控指标：可用性、错误率、核心转化路径。
2. 用户反馈分类与优先级（Bug、体验、需求；RICE 或同类框架）。
3. 固定迭代节奏：小版本快迭代，重大改动单独评审。
